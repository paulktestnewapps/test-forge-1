# Forge Contribute — Stage Local Changes for Upstream Contribution

Scans the current project for `.claude/` files that are new or have diverged from their forge-managed versions, then writes a `.forge-contributions.md` file describing the proposed contributions. This file is picked up by the GitHub Actions workflow that opens a PR against the central bunzl-forge repository.

## Step 1: Read Project Configuration

1. Read `.claude-config.yaml` — fail with a helpful message if missing ("Run /forge-init first")
2. Read `.forge-manifest.yaml` — fail if missing
3. Extract: stacks, project name, forge-managed file list with checksums

## Step 2: Locate the Bunzl Forge Repository

Find the bunzl-forge repo using the standard search:
1. `$HOME/projects/bunzl-forge`
2. `$HOME/repos/bunzl-forge`
3. `C:/repos/bunzl-forge` or `C:/projects/bunzl-forge`
4. `BUNZL_FORGE_PATH` environment variable

Needed to compute diffs for modified files.

## Step 3: Discover Contributable Changes

### New files (not in manifest)

Use Glob to find all `.md` files under `.claude/`:
```bash
find .claude/ -name "*.md" -type f
```

Cross-reference against `.forge-manifest.yaml`. Any file not present in the manifest is a **locally-added file** and a candidate for contribution.

### Modified forge-managed files

For each file in `.forge-manifest.yaml`:
1. Compute the current checksum: `sha256sum <path>` (Unix) or `Get-FileHash` (PowerShell)
2. Compare to the stored `checksum` in the manifest
3. If checksums differ, the file has been **locally modified**

### CLAUDE.md project-specific section

Read `CLAUDE.md` and extract the content inside the `<!-- bunzl-forge:section:project-specific -->` block. If it contains non-placeholder content (i.e., the user has filled it in), flag it as a candidate — portions of it may be worth promoting to the stack template.

## Step 4: Present Findings to User

Display a summary:

```
Forge Contribution Scan

New files (not tracked by forge):
  .claude/agents/my-new-agent.md
  .claude/skills/my-new-skill/SKILL.md

Modified forge-managed files:
  .claude/skills/analyze-endpoint/SKILL.md  (checksum mismatch)

CLAUDE.md project-specific section: has content
```

If nothing is found, report "No local changes detected that are candidates for contribution." and stop.

## Step 5: Gather Intent for Each Change

For each candidate, ask the user:

**For new files:**
Use AskUserQuestion for each file:
- "What does `<filename>` do, and why would it be useful to other teams?" (free text)
- "Which scope should this target in bunzl-forge?" with options:
  - `shared/.claude/` — Useful across all stacks
  - `stacks/<stack>/.claude/` — Specific to one stack (list stacks from config)
  - `Not sure` — Leave it for the bunzl-forge maintainer to decide
- "Any caveats or notes for the reviewer?" (free text, optional)

**For modified forge-managed files:**
Use AskUserQuestion:
- "Why did you modify `<filename>`? What problem does your change solve?" (free text)
- "Should this change replace the canonical version in bunzl-forge, or is it project-specific?" with options:
  - `Upstream it — replace the canonical version`
  - `Project-specific — just documenting it, not proposing a change`

**For the CLAUDE.md project-specific section:**
Use AskUserQuestion:
- "Does your CLAUDE.md project-specific section contain any patterns or conventions that would benefit other teams?" with options:
  - `Yes — include relevant excerpts in the contribution`
  - `No — it's project-specific, skip it`

Skip any item the user marks as not worth contributing.

## Step 6: Write `.forge-contributions.md`

Write this file to the project root. Overwrite if it already exists (current state only — not a rolling log).

Use this structure:

```markdown
# Forge Contributions

<!-- AUTO-GENERATED by /forge-contribute — edit descriptions only, do not change metadata -->

generated_at: <ISO timestamp>
project: <project name from .claude-config.yaml>
stacks: <stacks from .claude-config.yaml>

---

## Proposed Contributions

### <filename> — <new | modified>

- **Local path:** `.claude/<subdir>/<filename>`
- **Proposed target:** `<scope in bunzl-forge, or "TBD">`
- **Type:** `new` | `modified`
- **Description:** <user-provided description>
- **Caveats:** <user-provided caveats, or "None">

---

### <next file...>

---

## CLAUDE.md Excerpts

<!-- Only present if user opted to include project-specific section content -->

<extracted content from the project-specific block>

---

## Notes for Reviewer

<!-- Anything the user wanted to add that didn't fit per-file -->
<free-text notes, or "None">
```

Use one `###` block per contributed file. Preserve exact local paths so GH Actions can locate the files.

## Step 7: Confirm and Report

Tell the user:

```
.forge-contributions.md written.

Included: {N} new files, {N} modified files
Skipped:  {N} items (marked project-specific)

Next steps:
  1. Review .forge-contributions.md — you can edit descriptions directly
  2. Commit and push — the GitHub Actions workflow will open a PR to bunzl-forge
     with the proposed files and this changelog as context
```

Remind the user that `.forge-contributions.md` should be committed alongside the `.claude/` changes so the workflow can find both.
